# AWS EC2 — Elastic Compute Cloud

## 1. Core Concepts

### What is EC2?
EC2 provides **resizable virtual servers** (instances) in the cloud. You choose the OS, instance type, storage, and networking. AWS manages the physical hardware; you manage everything above the hypervisor.

### Instance Families
| Family | Optimized For | Example Types | Use Case |
|--------|--------------|---------------|----------|
| **T** (Burstable) | Balanced, low baseline + burst | t3.micro, t3.medium | Dev/test, small web apps |
| **M** (General) | Balanced CPU/memory | m6i.large, m6a.xlarge | Web servers, app servers |
| **C** (Compute) | High CPU | c6i.xlarge, c7g.large | Batch processing, ML inference |
| **R** (Memory) | High RAM | r6i.2xlarge | In-memory caches, large DBs |
| **P / G / Inf** (Accelerated) | GPU / ML | p4d.24xlarge, g5.xlarge | ML training, video rendering |
| **I / D** (Storage) | High I/O or Dense Storage | i4i.xlarge | NoSQL DBs, data warehousing |
| **X** (Memory Intensive) | SAP HANA scale | x2idn.32xlarge | In-memory analytics |

### Pricing Models
| Model | Description | Savings vs On-Demand | Best For |
|-------|-------------|----------------------|----------|
| **On-Demand** | Pay by second, no commitment | — | Unpredictable workloads, dev/test |
| **Reserved (RI)** | 1 or 3 year commitment, upfront/partial/no-upfront | Up to 72% | Steady-state production workloads |
| **Savings Plans** | Flexible commitment to spend ($/hr) | Up to 66% | Mixed instance types / Fargate |
| **Spot** | Bid on spare capacity, can be interrupted | Up to 90% | Fault-tolerant batch, Spark, CI/CD |
| **Dedicated Host** | Physical server reserved for you | Varies | Licensing (BYOL), compliance |
| **Dedicated Instance** | Instances on single-tenant hardware | Varies | Compliance without full host control |

### AMIs (Amazon Machine Images)
- Snapshot of an EC2 instance (OS, config, data volumes)
- **Public AMIs**: Amazon Linux 2, Ubuntu, Windows Server
- **Custom AMIs**: Bake your app into an AMI for fast, consistent scale-out
- **Marketplace AMIs**: Pre-configured third-party software (e.g., Fortinet, Bitnami)
- AMIs are **region-specific** — copy across regions using `CopyImage`

### Auto Scaling
- **Launch Template**: Defines instance config (AMI, type, SGs, user data)
- **Auto Scaling Group (ASG)**: Manages a fleet of instances
- **Scaling Policies**:
  - *Target Tracking* — maintain metric at a value (e.g., 50% CPU)
  - *Step Scaling* — scale in steps based on alarm thresholds
  - *Scheduled* — scale at known times (e.g., 9 AM peak)
  - *Predictive* — ML-based proactive scaling
- **Warm pools**: Pre-initialize instances to reduce launch latency

### Elastic Load Balancing (ELB)
| Type | OSI Layer | Protocol | Use Case |
|------|-----------|----------|----------|
| **ALB** (Application) | Layer 7 | HTTP/HTTPS/WebSocket | Web apps, microservices path routing, host-based routing |
| **NLB** (Network) | Layer 4 | TCP/UDP/TLS | Ultra-low latency, static IP, gaming, VoIP |
| **GWLB** (Gateway) | Layer 3+4 | GENEVE | Inline security appliances (IDS/IPS, firewall) |
| **CLB** (Classic) | Layer 4/7 | HTTP/HTTPS/TCP | Legacy — avoid for new deployments |

### Storage Options for EC2
| Type | Service | Persistence | Use Case |
|------|---------|-------------|----------|
| Block | EBS | Persistent | OS, databases |
| Block (temp) | Instance Store | Ephemeral | Cache, buffers, scratch |
| File | EFS | Persistent | Shared across instances |
| Object | S3 | Persistent | Artifacts, backups, static assets |

### Key Networking Concepts
- **Security Groups**: Stateful firewalls at the instance level (allow rules only)
- **Elastic IP**: Static public IPv4 address
- **ENI** (Elastic Network Interface): Virtual NIC — attach multiple ENIs to one instance
- **Placement Groups**: Control physical placement
  - *Cluster* — low latency, same AZ, same rack
  - *Spread* — high availability, different racks
  - *Partition* — distributed storage (HDFS, Cassandra)

## 2. Interview Questions & Answers

### Q1. What is the difference between a Security Group and a Network ACL?
**Security Groups** are **stateful** firewalls attached to ENIs. If you allow inbound traffic, return traffic is automatically allowed. They only support **allow** rules.  
**NACLs** are **stateless** firewalls attached to subnets. You must explicitly allow both inbound and outbound traffic. They support both **allow** and **deny** rules and are evaluated in numbered order.

**Rule of thumb:** Security Groups = per-instance defense; NACLs = subnet-level perimeter.

---

### Q2. What is the difference between On-Demand, Reserved, and Spot instances?
- **On-Demand**: No commitment, pay per second. Best for unpredictable workloads.
- **Reserved**: 1 or 3 year commitment in exchange for up to 72% discount. Best for steady-state production.
- **Spot**: Use AWS spare capacity at up to 90% discount, but AWS can reclaim with a 2-minute warning. Best for fault-tolerant, stateless workloads (batch, ML training, CI/CD).

---

### Q3. What happens when a Spot Instance is interrupted?
AWS sends a **Spot Instance interruption notice** 2 minutes before termination (via instance metadata and EventBridge). Best practices:
- Use **Spot interruption handler** (Lambda + EventBridge) to drain connections
- Use **Spot Fleet** or **EC2 Fleet** with multiple instance types/AZs to reduce interruption probability
- Offload state to S3/DynamoDB so work can be resumed

---

### Q4. How does EC2 Auto Scaling work? What is a Launch Template?
A **Launch Template** is a versioned EC2 configuration blueprint (AMI, instance type, key pair, SGs, user data). An **Auto Scaling Group** uses this template to launch and terminate instances based on demand.  
Scaling policies: Target Tracking (recommended), Step Scaling, Scheduled, and Predictive.

---

### Q5. What is the difference between vertical scaling and horizontal scaling in EC2?
- **Vertical Scaling (Scale Up/Down)**: Change instance type (e.g., t3.medium → m6i.xlarge). Requires stop/start, causes downtime unless using elastic resize or EBS.
- **Horizontal Scaling (Scale Out/In)**: Add/remove instances via Auto Scaling Groups. No downtime, better for stateless apps. Always preferred for production.

---

### Q6. What is an AMI and how do you use it for blue/green deployments?
An **AMI** is a pre-baked snapshot of an instance. For blue/green:
1. Build new AMI with updated code (v2)
2. Create a new Launch Template version pointing to the new AMI
3. Update the ASG to use the new template
4. Use an ALB listener rule to shift traffic 10% → 50% → 100% to the new ASG
5. Terminate old ASG on success, or roll back by redirecting traffic

---

### Q7. Explain EC2 Placement Groups. When would you use each type?
- **Cluster PG**: All instances on same rack in same AZ. Lowest latency, highest bandwidth (10 Gbps+). Use for HPC, ML training, tightly-coupled distributed workloads.
- **Spread PG**: Each instance on different hardware rack. Max 7 instances per AZ per group. Use for critical services needing high availability.
- **Partition PG**: Logical partitions on separate racks. Up to 7 partitions per AZ. Use for large distributed systems like HDFS, HBase, Cassandra.

---

### Q8. What is the difference between EBS and Instance Store?
| | EBS | Instance Store |
|--|-----|----------------|
| Persistence | Yes — survives stop/start | No — data lost on stop/terminate |
| Performance | Up to 256,000 IOPS (io2 Block Express) | Extremely high IOPS, NVMe local disk |
| Use Case | OS volumes, DBs | Temp buffers, caches, scratch data |
| Backup | Snapshots | Manual copy to S3 before termination |

---

### Q9. How does EC2 metadata work? What is IMDS v2?
Instance metadata is available at `http://169.254.169.254/latest/meta-data/`. It provides instance ID, AMI ID, IAM role credentials, etc.  
**IMDSv2** (recommended) requires a session-oriented approach using a PUT request first to get a token, preventing SSRF-based credential theft:
```bash
TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")
curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  http://169.254.169.254/latest/meta-data/instance-id
```

---

### Q10. What is the difference between ALB and NLB?
| | ALB | NLB |
|--|-----|-----|
| Layer | 7 (HTTP/HTTPS) | 4 (TCP/UDP/TLS) |
| Routing | Path, host, header, method | IP + Port |
| Latency | ~ms (slightly higher) | Ultra-low (<1ms) |
| Static IP | No (use AWS Global Accelerator) | Yes — per AZ |
| Use Case | Web apps, microservices | Gaming, VoIP, real-time, static IP requirement |

---

### Q11. How do you achieve zero-downtime deployments with EC2?
1. Use an **ALB** with multiple target groups
2. Deploy new instances to a second target group with the new version
3. Use **weighted target group routing** to shift traffic gradually (canary/linear)
4. Monitor CloudWatch metrics + alarms on error rates
5. Automate with **CodeDeploy** (in-place or blue/green strategies)

---

### Q12. What is EC2 User Data? How is it used in practice?
User Data is a shell script or cloud-init config that runs **once** at first boot as root. Used to:
- Install packages (`yum install nginx`)
- Pull config from SSM Parameter Store
- Bootstrap the app (pull from S3, configure, start service)
- Register with a service discovery system

---

### Q13. What are EC2 instance profiles and why are they important?
An **instance profile** attaches an **IAM role** to an EC2 instance. The instance automatically gets temporary credentials via the metadata service (IMDS). This eliminates the need to hard-code AWS credentials on the instance.  
Best practice: Grant **least-privilege** permissions through the role — only what the app needs.

---

### Q14. Explain EC2 Hibernate.
**Hibernate** saves the in-memory (RAM) state to the EBS root volume, then stops the instance. On restart, the RAM is restored instantly, so the OS and applications resume from exactly where they left off.  
Useful for long-running in-memory processes, ML model serving warm-up, or costly application initialization.

---

### Q15. How would you design a highly available, fault-tolerant web application on EC2?
1. **Multi-AZ**: Deploy ASG across at least 2 AZs
2. **ALB**: Distribute traffic, with health checks
3. **Auto Scaling**: Target tracking policy on CPU or ALB request count
4. **RDS Multi-AZ**: Standby in second AZ
5. **ElastiCache**: Session/cache layer (not stored on EC2)
6. **S3**: Static assets, user uploads
7. **Route 53**: DNS failover + latency routing
8. **CloudWatch Alarms**: Trigger scaling and alert on anomalies

## 3. Real-World Use Case: E-Commerce Web Tier

### Scenario
A retail company needs a web tier that:
- Handles 10,000 concurrent users normally, spikes to 80,000 on sale days
- Requires < 200 ms response time
- Must have 99.9% availability
- Needs to pass PCI-DSS compliance for payment data

### Architecture

```
Internet
   │
[Route 53] ← Health checks + failover routing
   │
[AWS WAF] ← Block SQLi, XSS, rate limiting
   │
[ALB] (Multi-AZ, HTTPS listener, SSL termination)
   │
┌──────────────────────────────────────────┐
│          Auto Scaling Group               │
│  ┌─────────────┐    ┌─────────────┐      │
│  │ EC2 (AZ-a)  │    │ EC2 (AZ-b)  │      │
│  │ m6i.xlarge  │    │ m6i.xlarge  │      │
│  │ Amazon Linux│    │ Amazon Linux│      │
│  │ min:4 max:40│    │             │      │
│  └──────┬──────┘    └──────┬──────┘      │
└─────────┼─────────────────┼─────────────┘
          │                 │
    ┌─────▼─────────────────▼─────┐
    │     ElastiCache (Redis)      │
    │     Session store + cache    │
    └─────────────┬───────────────┘
                  │
    ┌─────────────▼───────────────┐
    │   RDS Aurora MySQL          │
    │   Multi-AZ + Read Replicas  │
    └─────────────────────────────┘
          │
    [S3 + CloudFront]  ← Static assets, images
```

### Key Design Decisions

| Decision | Why |
|----------|-----|
| `m6i.xlarge` instances | Balanced CPU/RAM for PHP/Node.js app; avoid T-type for sustained production load |
| Spot instances for batch | Nightly report generation on Spot Fleet — 70% cheaper |
| Target tracking at 60% CPU | Leaves headroom for spikes; ASG launches at 70%+ before it's needed |
| Reserved instances (1-year) | Baseline 4 instances always running — ~50% savings |
| ALB + WAF | Layer 7 protection + SSL termination in one place |
| Custom AMI | Pre-baked with app code, reduces launch time from 8 min → 90 sec |
| IMDSv2 enforced | Prevents SSRF credential theft — required for PCI-DSS |

### How to Architect This for a $100 Budget (Practice Lab)
1. Launch 1 × `t3.medium` (free tier or ~$0.04/hr)
2. Create an ALB ($0.008/hr + LCU)
3. Create an ASG with min=1, max=3
4. Use Apache Bench or k6 to load test and watch scaling kick in
5. Terminate all resources when done — total cost ~$2-3 for a 4-hour lab

### Interview Narration (White-board Script)
> "For this e-commerce platform I'd start with an ALB to distribute traffic across an Auto Scaling Group of EC2 instances spread across two AZs. I'd use custom AMIs to keep launch times under two minutes. The ASG uses target tracking at 60% CPU so it starts scaling before we hit saturation. Sessions are stored in ElastiCache Redis — not on the instance — so any instance can handle any request. Static assets go to S3 and are served via CloudFront to reduce load on origin servers. The database is Aurora MySQL Multi-AZ for automatic failover under 30 seconds. For cost optimization, the baseline fleet uses Reserved Instances and we burst with On-Demand or Spot for batch jobs."
