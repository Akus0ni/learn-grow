# Core Capabilities of an AI Agent

> What separates an AI Agent from a plain LLM is a set of capabilities that let it *act* in the world, not just *respond* to it.

The four foundational capabilities — plus three more that matter in production systems.

## 1. Planning

> **Breaks goals into tasks autonomously**

Planning is the ability to take a high-level, often ambiguous goal and decompose it into a concrete sequence of steps that can be executed one by one.

### What it involves
- **Goal decomposition** — split "research competitors and send a report" into 8 concrete sub-tasks
- **Dependency ordering** — "I need to fetch data before I can analyze it"
- **Conditional branching** — "if the API fails, fall back to web search"
- **Replanning** — adapt the plan mid-execution when something unexpected happens

### How it works in practice

```
Goal: "Write a blog post about the top 5 AI tools of 2025"

Plan generated by agent:
  Step 1: Search web for "best AI tools 2025"
  Step 2: Filter to top results from credible sources
  Step 3: For each tool — fetch name, description, key features
  Step 4: Rank by popularity / relevance
  Step 5: Draft blog post with intro, tool sections, conclusion
  Step 6: Review draft for tone and accuracy
  Step 7: Return final post
```

No human specified these steps. The agent derived them from the goal.

### Planning Techniques

| Technique | Description | When to use |
|---|---|---|
| **Chain-of-Thought (CoT)** | Step-by-step reasoning in a single prompt | Simple linear tasks |
| **ReAct** | Reason → Act → Observe loop | Tasks with tool calls |
| **Tree-of-Thought (ToT)** | Explore multiple plan branches, pick best | Complex decisions |
| **Plan-and-Solve** | Explicit plan first, then execute | Long multi-step tasks |
| **MCTS** | Monte Carlo Tree Search — explore & evaluate paths | Game-like optimization |
| **HuGE / BabyAGI** | Agent creates and manages its own task queue | Open-ended autonomous tasks |

### Tools & Frameworks for Planning

| Tool | Role |
|---|---|
| `LangGraph` | Build explicit plan graphs with conditional edges |
| `AutoGen` | Multi-agent planning via conversation |
| `CrewAI` | Assign roles to agents, each with a sub-plan |
| `TaskWeaver` (Microsoft) | Code-first planning for data analytics tasks |
| `Llama Index Workflows` | Event-driven planning pipelines |
| `Dust` | Visual agent workflow builder |

## 2. Memory & Knowledge

> **Stores and retrieves information**

Memory is what lets an agent learn from experience, maintain context across sessions, and make personalized, informed decisions — rather than starting from scratch every time.

### The Four Types of Agent Memory

```
┌──────────────────────────────────────────────────────────────┐
│                     AGENT MEMORY                             │
│                                                              │
│  ┌─────────────────┐      ┌────────────────────────────┐    │
│  │   IN-CONTEXT    │      │      EXTERNAL STORAGE      │    │
│  │  (Short-term)   │      │       (Long-term)          │    │
│  │                 │      │                            │    │
│  │ Current thread  │      │  ┌──────────┬──────────┐  │    │
│  │ Tool results    │      │  │ Semantic │ Episodic │  │    │
│  │ Recent history  │      │  │ (facts)  │ (events) │  │    │
│  └─────────────────┘      │  └──────────┴──────────┘  │    │
│                            │  ┌──────────────────────┐ │    │
│                            │  │  Procedural (skills) │ │    │
│                            │  └──────────────────────┘ │    │
│                            └────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

| Memory Type | What It Stores | Example |
|---|---|---|
| **Short-term (in-context)** | Current conversation, recent tool results | Messages in the chat window |
| **Semantic** | General facts, user preferences, domain knowledge | "User prefers bullet-point summaries" |
| **Episodic** | Specific past events and outcomes | "On March 3rd, the Stripe API call failed — used fallback" |
| **Procedural** | How to do things — skills and workflows | "Steps to generate a weekly report" |

### Memory in Action

```
Session 1:
  User: "I'm building a SaaS for HR teams. My stack is Next.js + Supabase."
  Agent: [saves to long-term memory: user → SaaS, domain → HR, stack → Next.js + Supabase]

Session 2 (one week later):
  User: "How should I structure my database for multi-tenancy?"
  Agent: [recalls stack is Supabase (PostgreSQL)]
  → "For your Next.js + Supabase setup, the cleanest multi-tenant pattern
     is Row-Level Security (RLS) with a tenant_id column..."
```

No re-introduction needed. The agent remembered.

### Knowledge Retrieval — RAG

**Retrieval-Augmented Generation (RAG)** lets agents query private knowledge bases:

```
User: "What does our refund policy say about digital products?"

Without RAG: LLM guesses or says "I don't have that information"

With RAG:
  1. Embed user query → search vector DB of company docs
  2. Retrieve top-3 relevant policy chunks
  3. LLM reads chunks + generates accurate, cited answer
```

## 3. Tool Interaction

> **Uses external tools and APIs**

Tools are the agent's hands. Without tools, the agent can only *think*. With tools, it can *act* on any system in the world.

### What a Tool Is

A tool is any callable function that the agent can invoke:
- A web search
- A database query
- A code execution environment
- An HTTP API call
- A file operation

The agent's LLM decides *which* tool to call, *when*, and *with what arguments* — just from reading the tool's description.

### Tool Call Flow

```
Agent thinks: "I need the current weather in London"

Tool available: get_weather(city: str) → Returns current weather

Agent generates:
  {
    "tool": "get_weather",
    "arguments": { "city": "London" }
  }

Tool executes → Returns: "London: 12°C, partly cloudy, wind 15 km/h"

Agent reads result → Incorporates into response
```

## 4. Action Execution

> **Performs tasks independently**

Action Execution is what makes an agent *useful* — it's the difference between an agent that plans and one that actually *does*.

### Types of Actions

| Action Type | Description | Example |
|---|---|---|
| **Read** | Retrieve information without changing state | Search web, query DB, read file |
| **Write** | Modify state in an external system | Update DB, write file, create ticket |
| **Communicate** | Send messages or notifications | Send email, post Slack message |
| **Compute** | Process or transform data | Run Python, execute SQL, do math |
| **Spawn** | Create sub-agents or sub-processes | Launch a research agent, start a job |
| **Control** | Operate interfaces or systems | Click button in browser, run shell cmd |

### Action Execution Loop

```
┌─────────────────────────────────────────────────────┐
│                                                     │
│   Goal Set                                          │
│       ↓                                             │
│   Agent Plans Steps                                 │
│       ↓                                             │
│   ┌───────────────────────────────┐                 │
│   │  For each step:               │                 │
│   │    1. Select tool             │                 │
│   │    2. Prepare arguments       │                 │
│   │    3. Execute action          │                 │
│   │    4. Observe result          │                 │
│   │    5. Update plan if needed   │                 │
│   └───────────────────────────────┘                 │
│       ↓                                             │
│   Goal Complete → Return result                     │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### Safe Action Execution Patterns

Because agents take *real actions with real consequences*, production systems use these patterns:

| Pattern | Description |
|---|---|
| **Human-in-the-loop** | Pause before high-stakes actions for human approval |
| **Dry run mode** | Simulate actions without executing them (show what *would* happen) |
| **Sandboxing** | Code runs in isolated container — can't affect host system |
| **Rollback** | Keep state snapshots so actions can be undone |
| **Budget limits** | Max API calls, spend caps, time limits |
| **Scope restriction** | Agent can only act on specific resources it's been granted |

### Tools for Action Execution

| Tool | Role |
|---|---|

| `Anthropic Tool Use` | Claude-native tool definition and execution |

## 5. Reasoning & Decision Making

> **Choosing the right action at the right time**

Often bundled with planning, but distinct: reasoning is the *moment-to-moment* judgment the agent applies. This is where the LLM's intelligence shines.

### What Reasoning Covers

- **Tool selection:** Given 10 available tools, which one should I use right now?
- **Ambiguity resolution:** The user said "send it to the team" — which team, which channel?
- **Trade-off evaluation:** Should I use the fast/cheap API or the accurate/expensive one?
- **Risk assessment:** This action is irreversible. Should I confirm with the user first?
- **Prioritization:** I have 5 sub-tasks. Which is most critical to do first?

### Reasoning Patterns

| Pattern | How it works | Example |
|---|---|---|
| **Chain-of-Thought** | "Let me think step by step..." | Math problem solving |
| **ReAct** | Think → Act → Observe → Think... | Tool-using agents |
| **Self-Ask** | Breaks question into sub-questions | Research tasks |
| **Least-to-Most** | Solve simplest sub-problem first, build up | Complex decomposition |
| **Socratic** | Asks clarifying questions before acting | Ambiguous requests |

### Tools for Better Reasoning

| Tool | Role |
|---|---|
| `DSPy` | Optimize prompts for better reasoning automatically |
| `Guidance` | Control LLM output structure during reasoning |
| `Outlines` | Constrained generation for structured outputs |
| `Instructor` | Force LLM to output valid typed objects |

## 6. Reflection & Self-Correction

> **Learns from mistakes within a task**

Reflection is the agent's ability to step back, evaluate what it did, identify errors, and improve — without human intervention.

### Without Reflection

```
Agent calls API → Gets error → Crashes or returns wrong answer
```

### With Reflection

```
Agent calls API → Gets error
  → Reflects: "I passed a string but the API expects an integer"
  → Corrects argument type
  → Retries successfully
```

### Types of Reflection

| Type | Description |
|---|---|
| **In-task correction** | Fixes mistakes within the current execution |
| **Post-task critique** | Reviews completed output for quality before returning |
| **Cross-task learning** | Stores lessons from past tasks to improve future ones |
| **Peer review** | Another agent reviews and critiques the output |

### Real Example — Code Debugging Loop

```
Agent writes Python function
  → Runs it
  → Error: "TypeError: unsupported operand type(s) for +: 'int' and 'str'"
  → Reflects on error message
  → Identifies line 7 is concatenating int + str
  → Fixes: str(count) + " items"
  → Reruns
  → Tests pass ✓
  → Returns working code to user
```

### Tools for Reflection

| Tool | Role |
|---|---|
| `Reflexion` (technique) | Agent writes reflection notes after each task failure |
| `AutoGen` | Agents critique each other's outputs |
| `LangGraph` | Build explicit retry/reflection edges in agent graph |
| `CRITIC` (technique) | Agent uses tools to verify and critique its own claims |
| `Constitutional AI` | Built-in self-critique based on a set of principles |

## 7. Multi-Agent Coordination

> **Orchestrates specialist agents to tackle complex problems**

Some tasks are too large or specialized for one agent. Multi-agent systems split work across agents with different roles, tools, or expertise.

### Patterns

#### Orchestrator → Workers
```
Orchestrator Agent
    ├── Research Agent     (searches web, reads papers)
    ├── Analysis Agent     (runs Python, crunches numbers)
    ├── Writing Agent      (drafts content)
    └── Review Agent       (checks facts, tone, errors)
```

#### Peer Debate
```
Agent A writes a solution
Agent B critiques it
Agent A refines based on critique
→ Higher quality output than either alone
```

#### Parallel Execution
```
Task: "Research 5 competitors simultaneously"

Agent 1 → researches Competitor A  ─┐
Agent 2 → researches Competitor B   │
Agent 3 → researches Competitor C   ├──► Aggregator → Final Report
Agent 4 → researches Competitor D   │
Agent 5 → researches Competitor E  ─┘
```

### Tools for Multi-Agent Coordination

| Tool | Role |
|---|---|
| `AutoGen` (Microsoft) | Conversation-based multi-agent framework |
| `CrewAI` | Role-based agent teams with defined goals |
| `LangGraph` | Explicit graph edges between agents |
| `Anthropic Agent SDK` | Spawn and orchestrate sub-agents natively in Claude |
| `AgentVerse` | Research framework for large-scale agent simulations |
| `SuperAGI` | Open-source multi-agent execution platform |

## Full Capability Summary

| Capability | Core Function | Without It | Tools |
|---|---|---|---|
| **Planning** | Goal → Steps | Agent doesn't know what to do | LangGraph, CrewAI, AutoGen |
| **Memory & Knowledge** | Store & retrieve | Forgets everything, starts fresh each time | Pinecone, mem0, LlamaIndex |
| **Tool Interaction** | Use external systems | Can only generate text | Composio, LangChain Tools, Zapier |
| **Action Execution** | Do things in the world | Plans but never acts | E2B, Docker, Anthropic Tool Use |
| **Reasoning & Decision** | Choose what to do next | Makes random/bad choices | DSPy, Instructor, Outlines |
| **Reflection** | Fix mistakes autonomously | Fails silently or returns wrong results | Reflexion, AutoGen, LangGraph |
| **Multi-Agent Coordination** | Scale via specialization | Bottlenecked to one agent's capability | CrewAI, AutoGen, LangGraph |

## The Minimum Viable Agent

You don't need all 7 capabilities for every use case. Here's what each level requires:

```
Level 1 — Chatbot with tools
  ✓ Tool Interaction
  ✓ Reasoning

Level 2 — Task Agent
  ✓ Planning
  ✓ Tool Interaction
  ✓ Action Execution
  ✓ Reasoning

Level 3 — Autonomous Agent
  ✓ All of Level 2
  ✓ Memory & Knowledge
  ✓ Reflection & Self-Correction

Level 4 — Multi-Agent System
  ✓ All of Level 3
  ✓ Multi-Agent Coordination
```

The more capabilities you add, the more autonomous and powerful — but also the more complex to build, debug, and trust.
